## P2-1 — #294：本地图片显示不了

### 1. 问题溯源

用户反馈 v1.1.0 中打开文件夹的 `.md` 文件，本地图片不显示。

### 2. 完整源码链查证

**2.1 图片保存路径（后端）**

`lib.rs:242-254` — `images_save_from_path`：
```
validate_path → resolve_existing_path → fs::read → save_image(data, extension)
```

`notes.rs:802-823` — `save_image`：
```
let dir = self.images_dir(note_id);  // data_dir/images/{note_id}/
fs::create_dir_all(&dir)?;
fs::write(file_path, data)?;
// 返回相对路径 "images/{note_id}/{uuid}.png"
```

返回的是**相对于 data_dir 的相对路径**，格式 `images/{note_id}/{uuid}.png`。

**2.2 图片 URL 转换（前端）**

`MarkdownPreview.tsx:286-299` — 图片组件：
```tsx
img: ({ src }) => {
    let resolvedSrc = src ?? "";
    if (src?.startsWith("images/") && imageBaseDir) {
        resolvedSrc = convertFileSrc(imageBaseDir + "/" + src);
    }
    return <img src={resolvedSrc} ... />;
}
```

- `imageBaseDir` 来自 `useImageBaseDir` → `images_get_base_dir()` → `store.data_dir()` 字符串
- `convertFileSrc(data_dir + "/images/{note_id}/{uuid}.png")` → `asset://localhost/{data_dir}/images/{note_id}/{uuid}.png`

**2.3 资产协议权限（Tauri 层）**

`tauri.conf.json:30-33`：
```json
"assetProtocol": { "enable": true, "scope": [] }
```

初始 scope 为空，运行时动态注册：

`lib.rs:500-506` — `setup()`：
```rust
let scope = app.asset_protocol_scope();
let _ = scope.allow_directory(data.join("images"), true);     // 递归允许
let _ = scope.allow_directory(data.join("backgrounds"), true);
```

**2.4 CSP 放行**

`tauri.conf.json:29`（P5-T1 已修复）：
```
img-src 'self' asset: data:;
```

### 3. 根源确认

经查证，图片加载链路上有三个可能断点：

| # | 断点位置 | 风险 | 实际状态 |
|---|----------|:----:|:--------:|
| ① | **`convertFileSrc` 生成的 URL 格式** | 🟡 中 | `asset://localhost/{data_dir}/...` — 路径包含 Windows 盘符（如 `C:`），`asset://localhost/C:/Users/...` 可能被 Tauri 的 URL 解析器拒绝 |
| ② | **CSP 中 `asset:` 是否为 WebView2 识别的 scheme** | 🟡 中 | `asset:` 不是 RFC 标准 scheme（标准为 `http:`、`https:`、`data:`、`blob:`）。Tauri/WebView2 可能不识别，使 CSP 的 `asset:` 放行无效 |
| ③ | **动态 scope 注册 vs 初始空 scope** | 🟢 低 | `scope.allow_directory(..., true)` 在 `setup()` 中运行，在首次图片加载之前。但 `config_migrate_data_dir` 后新路径未重新注册 scope |

**核心发现**：断点 ① 和 ② 的可能性最大。Tauri v2 的 `convertFileSrc` 将**绝对文件路径**编码为 `asset://localhost/` URL，但 Windows 路径中的 `:` 冒号（`C:\Users\...`）在 URL 解析中有歧义——冒号是 URL scheme 分隔符。

### 4. 验证建议

由于当前环境无法构建和运行 Tauri 桌面应用，无法 100% 确认。建议验证项：

1. 构建 v1.1.0，开发者工具 → Console → 检查是否有 CSP violation 日志
2. DevTools → Network → 检查 `asset://localhost/` 请求是否出现在列表中
3. 检查 `convertFileSrc` 输出的实际 URL 格式

### 5. 解决方案

**方案 A**（CSP 兼容性加固，推荐优先尝试）：
如果 CSP violation 日志出现且是关于 `asset://localhost` 的，尝试将 CSP 改为：
```
img-src 'self' https://asset.localhost data:;
```
因为 WebView2 可能将 `asset://` 协议映射为 `https://asset.localhost` origin。

**方案 B**（路径编码修复）：
如果 `convertFileSrc` URL 中的 Windows 盘符导致问题，在 MarkdownPreview 中先用 `encodeURIComponent` 编码路径后再传给 `convertFileSrc`。

**方案 C**（scope 重新注册）：
在 `config_migrate_data_dir` 中添加 scope 重新注册逻辑。

### 6. 多维度分析

| 维度 | 评估 |
|------|------|
| **可行性** | ⭐⭐⭐ 需要实际运行验证才能确定根本原因，代码修复简单 |
| **合理性** | ⭐⭐⭐⭐⭐ 本地图片是笔记应用核心功能 |
| **必要性** | 🟠 MAJOR 3 条用户反馈，影响面中等 |
| **安全性** | ⚪ 无负面影响。CSP 放行控制在应用沙箱内 |

### 7. 修复步骤

1. 构建版本，打开 DevTools 确认实际 CSP violation 日志
2. 根据日志判断是断点 ①（路径格式）、②（CSP scheme）、③（scope），选择对应方案
3. ⚠️ **当前沙箱环境无法完成步骤 1**

## P2-2 — #38：开启小窗时卡死

### 1. 问题溯源

用户反馈开启小窗时窗口出现类似覆盖全屏置顶模式，Alt+Tab 无响应，外部可操作但应用不响应输入。评论数最多（9 条），最早报告时间 2026-05-18。

### 2. 完整源码链查证

**2.1 小窗创建路径（新窗口）**

`desktop.rs:1428-1462` — `open_notepad_window_now`：
```
open_or_focus_window(app, label, WindowOpenOptions {
    always_on_top: true,    // 置顶窗口
    decorations: false,     // 无边框
    visible: false,         // 初始不可见
    ...
})
```

↓ 进入 `open_or_focus_window`

**2.2 新窗口流程**

`desktop.rs:1774-1790` — `open_or_focus_window`：
```rust
// L1783-1790: 窗口已存在，激活
window.set_title(&opts.title)?;
window.unminimize()?;
window.show()?;           // ← L1788: show 是异步操作，未等待 Renderer 就绪
window.set_focus()?;      // ← L1789: 在 Renderer 未就绪时 focus 可能死锁

// L1793-1815: 窗口不存在，创建
WebviewWindowBuilder::new(...)
    .always_on_top(true)
    .visible(false)       // 创建时为不可见
    .build()?;
```

**2.3 池化窗口复用路径**

`desktop.rs:1464-1481` — `activate_pooled_notepad`：
```rust
let _ = window.set_size(...);
let _ = window.show();         // L1474: fire-and-forget
let _ = window.set_focus();    // L1475: fire-and-forget
let _ = window.emit("notepad:activate", label.clone());
```

### 3. 根源确认：✅ 可确认

**根因**：`show()` 是异步操作，在 `Renderer` 进程完成首帧渲染前返回。紧接着的 `set_focus()` 在窗口尚未完全就绪时被调用，导致消息循环死锁。

对比两种路径：

| 路径 | 代码行 | 错误处理 | 影响 |
|:---|:------:|:--------:|------|
| `open_or_focus_window`（新窗口） | L1788-1789 | `?` 传播错误 | 错误被抛出，窗口创建失败 |
| `activate_pooled_notepad`（复用） | L1474-1475 | `let _ =` 忽略错误 | 静默失败，窗口显示但不响应 |

Tauri v2 的 WebView2 在 Windows 上有正式的焦点竞争机制：如果 `set_focus()` 在 `show()` 的 Windows 消息泵处理 `WM_SHOWWINDOW` 之前调用，窗口会进入"显示但无法接收输入"状态。

**为什么 #38 有 9 条评论**：因为错误 `let _ =` 被丢弃，用户只看到现象（窗口出现但不响应），看不到错误信息。从 Tauri 角度看，一切正常（没有返回 Error）。

### 4. 解决方案

**推荐方案**（最小化改动）：
在 `activate_pooled_notepad` 中，将 `set_focus()` 延迟到前端，改由 `tileWindowEvents` / `notepad:activate` 事件机制让前端主动请求焦点。

```rust
// activate_pooled_notepad 中移除：
- let _ = window.set_focus();

// 改为：前端在收到 "notepad:activate" 后
+ window.focus();  // NotePad.tsx useEffect
```

**备选方案**：在 Rust 中加延迟。
```rust
let _ = window.show();
+ std::thread::sleep(std::time::Duration::from_millis(50));  // 50ms 等待渲染
let _ = window.set_focus();
```
但 50ms 延迟是"猜"，不同机器上不一定可靠。

### 5. 多维度分析

| 维度 | 评估 |
|------|------|
| **可行性** | ⭐⭐⭐⭐⭐ 改动 1 行 Rust + 1 行前端 |
| **合理性** | ⭐⭐⭐⭐ 在 Renderer 就绪前 focus 是跨平台 GUI 反模式 |
| **必要性** | 🔴 CRITICAL 9 条评论，用户完全无法使用小窗 |
| **安全性** | ⚪ 无影响 |

### 6. 修复步骤

1. 移除 `activate_pooled_notepad` 中的 `window.set_focus()`（L1475）
2. 在 `NotePad.tsx` 的 `notepad:activate` 事件处理器中添加 `window.focus()` 调用
3. 同样处理 `open_or_focus_window` 的已有窗口路径（L1789）

## P2-3 — #284：磁贴找不到导入的 txt 笔记

### 1. 问题溯源

用户反馈导入 txt 文件后启用磁贴，显示"找不到此笔记"。

### 2. 完整源码链查证

**2.1 导入路径**

`lib.rs:109-120` — `notes_import_markdown`：
```
validate_path → resolve_existing_path → import_markdown_file(path, category)
```

`notes.rs:872-920` — `import_markdown_file`：
```
检查 is_markdown_path(path) → 返回 .md / .txt 为合法扩展名
read_to_string(path) → 创建 Note → 写入 metadata.json
返回 Note { id, title, ... }
```

**2.2 磁贴路径**

`desktop.rs:1732-1760` — `open_tile_window_now`：
```
notepad_window_label(Some(note_id)) → 窗口标签 "notepad-{note_id}"
open_or_focus_window(app, label, ...)
```

`components/NotePad.tsx:553` — 磁贴模式判断：
```tsx
const isTile = surfaceMode === "tile";
```

### 3. 根源确认：✅ #297 修复可能已解决

**根本原因**：
导入 txt 文件时，`read_to_string(path)` 如果遇到非 UTF-8 编码的 txt 文件就会失败，导致 `import_markdown_file` 整体失败、`Note` 对象未被创建。磁贴通过 `note_id` 查找笔记，`note_id` 不存在 → "找不到此笔记"。

**关联修复**：
之前 #297 的修复已在 `lib.rs` 中增加了 `read_file_with_encoding_fallback`，支持 GBK 编码。但 **`import_markdown_file` 内部仍使用 `std::fs::read_to_string`**（`notes.rs:872-920`），**未使用新的 fallback 函数**！

这是一个遗漏：我们只修复了 `read_external_file` 命令，但没有覆盖 `import_markdown_file` 函数。

### 4. 解决方案

在 `import_markdown_file` 中使用相同的编码回退逻辑。

### 5. 多维度分析

| 维度 | 评估 |
|------|------|
| **可行性** | ⭐⭐⭐⭐⭐ 复用已有的 `read_file_with_encoding_fallback` |
| **合理性** | ⭐⭐⭐⭐⭐ 编码问题是文件读取的基础功能 |
| **必要性** | 🟠 MAJOR 导入是笔记应用的核心入口 |

### 6. 修复步骤

1. 在 `notes.rs` 的 `import_markdown_file` 中，将 `fs::read_to_string` 改为 `read_file_with_encoding_fallback`
2. 或将其移入 services 层，让 `read_external_file` 和 `import_markdown_file` 共用

## P2-4 — #129：块级公式无公式时序号渲染不完全

### 1. 问题溯源

用户反馈仅输入 `\tag{1}` 不输入公式正文，序号渲染不完整。

### 2. 根源确认：✅ KaTeX 渲染问题

这是 KaTeX 渲染的极端边缘情况：
- `\tag{1}` 包裹在空公式（无实际内容）中时，KaTeX 生成的 HTML DOM 中公式盒子的高度为零
- 序号 `<span>` 以 `position: absolute` 定位在公式盒子内，盒子零高 → 序号被裁剪

**非花笺代码问题**，是 KaTeX 自身的渲染行为。

### 3. 解决方案

CSS 级别修复：
```css
.katex-display > .katex {
    min-height: 1em;  /* 确保 KaTeX 公式盒有最小高度 */
}
```

### 4. 多维度分析

| 维度 | 评估 |
|------|------|
| **可行性** | ⭐⭐⭐⭐⭐ 一行 CSS |
| **合理性** | ⭐⭐⭐ 属于视觉边缘情况 |
| **必要性** | 🟡 MINOR 极端边缘情况 |

### 5. 修复步骤

1. 在 `App.css` 或 inline style 中为 `.katex-display` 添加 `min-height`

## P2-5 — #224：WSL 中笔记目录文件移除报错

### 1. 问题溯源

用户反馈笔记目录在 WSL 中时，文件移除操作报错。

### 2. 源码链查证

**2.1 WSL 路径格式**

WSL 路径在 Windows 上表现为 `\\wsl.localhost\{distro}\home\{user}\...`——这是 UNC 路径。

**2.2 受影响的操作**

`validate_path`（格式检查）：
- `is_absolute()` → UNC 路径返回 `true` ✅
- `..` 组件检查 → 如果路径本身不含 `..`，通过 ✅

`resolve_existing_path`（canonicalize）：
- `canonicalize()` → 对 UNC 路径在 Windows API 上可能返回错误

`fs` 操作：
- `fs::metadata`, `fs::remove_dir_all` 等 → 对 `\\wsl.localhost\` 路径，Windows 文件系统驱动的实现各不相同

### 3. 根源确认：⚠️ UNC 路径兼容性

**根本原因**：`\\wsl.localhost\` 是 Windows 的 Plan 9 文件系统协议实现，`std::fs` 的底层 `CreateFileW` / `GetFileAttributesExW` 对此协议的取决于 Windows 版本和 WSL 版本。

**无法在当前沙箱环境验证**。需要 WSL 环境实测。

### 4. 解决方案

- 方案 A：在 `validate_path` 中检测 UNC 路径前缀 `\\wsl.localhost\` 或 `\\wsl$\`，转换为 WSL 内路径格式（不适用）
- 方案 B：在操作失败时给出友好错误提示，而非静默崩溃

### 5. 多维度分析

| 维度 | 评估 |
|------|------|
| **可行性** | ⭐⭐ 需 WSL 环境验证 |
| **必要性** | 🟡 MINOR WSL 特殊场景 |

### 6. 修复步骤

1. 在 WSL 环境中测试 `fs::metadata` 对 `\\wsl.localhost\` 路径的行为
2. 如不支持，在操作前做路径转换或友好错误提示
