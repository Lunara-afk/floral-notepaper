### L1 — T3 遗留：默认预览模式无 HTML sanitize

**源码查证** ✅ 已完成

`MarkdownPreview.tsx:87-93`：
```typescript
const rehypePluginsDefault = [rehypeKatex, rehypeSlug];     // ❌ 无 sanitize
const rehypePluginsWithHtml = [
  rehypeRaw, [rehypeSanitize, sanitizeSchema], rehypeKatex, rehypeSlug  // ✅ 有 sanitize
];
```

当 `renderHtml=false`（默认），Markdown 内容**不经 `rehypeSanitize` 处理**。`react-markdown` 将 raw HTML 作为危险跳过（默认安全），但某些边缘情况（Markdown 解析器允许的 HTML 标签）可能透传。

**解决方案**：为 `rehypePluginsDefault` 添加 `[rehypeSanitize, sanitizeSchema]`，与 html 模式使用相同的 sanitize 配置

**影响分析**：
- 安全性：阻断默认模式下可能的 HTML 注入
- 功能：如果用户依赖 raw HTML 渲染（默认模式下），会被 sanitize 剥离。但默认模式本就预期不渲染 HTML
- 与 T3 的关系：T3 仅影响 `rehypePluginsWithHtml` 路径。此修复补齐默认模式的防护

**可行性** ⭐⭐⭐⭐⭐：一行数组修改
**合理性** ⭐⭐⭐⭐⭐：两套路径使用同一 sanitize 配置是正确的分层防御
**必要性** ⭐⭐⭐：CSP 已提供顶层屏蔽，但 sanitize 是第二道防线。安全问题审查标记为 MAJOR 缺口

**修复步骤**：
1. 修改 `MarkdownPreview.tsx:87`，替换 `rehypePluginsDefault` 为包含 sanitize 的配置
2. 测试默认模式下 GFM alerts、KaTeX 渲染不受影响
3. 测试默认模式下嵌入 `<script>`、`<style>` 标签被剥离

### L2 — T8 遗留：useImagePaste dispatchEvent 模式重构

**源码查证** ✅ 已完成

`useImagePaste.ts:46-61` — `insertTextAtCursor` 函数：
```typescript
textarea.value = before + insertion + after;              // 直接 DOM 操作
textarea.dispatchEvent(new Event("input", { bubbles: true })); // 欺骗 React
setContent(textarea.value);                               // 同步 React state
```

`dispatchEvent("input")` 是一种"欺骗 React 受控组件"的模式——直接修改 DOM 但通知 React 数据已变。虽然功能正常，但与 React 单向数据流设计不符。

**解决方案**：将插入逻辑提升到 `NotePad` 组件中，通过 prop 回调统一管理文本插入，在 React state 层完成文本拼接而非 DOM 层

**影响分析**：
- 架构：React state → DOM 的单向数据流回归，消除 DOM 直接操作
- 功能：无变化
- 风险：低（当前方案已工作，重构价值在可维护性）

**可行性** ⭐⭐⭐⭐：需重构 `NotePad.tsx` 中 `insertTextAtCursor` 的调用方
**合理性** ⭐⭐⭐⭐⭐：符合 React 最佳实践
**必要性** ⭐⭐：当前方案功能正常，不导致 bug。价值在代码清洁度和未来可维护性

**修复步骤**：
1. 在 `NotePad` 中新增 `handleInsertText(text: string)` 方法，基于当前 `content` state 拼接文本
2. 将 `insertTextAtCursor` 改为纯函数，不再操作 DOM（仅计算新的 cursor 位置和文本片段）
3. `useImagePaste` 和 `MainWindow` 通过 prop 回调调用统一的文本插入逻辑

| 字段 | 值 |
|------|------|
| **上报人** | Vallily |
| **提交时间** | 2026-05-18（最早 bug） |
| **版本** | 未注明 |
| **评论** | 9（最多讨论） |
| **平台** | Windows |
| **复现描述** | 小窗出现类似覆盖全屏置顶模式，Alt+Tab 无响应，外部可操作但应用不响应输入 |

**源码查证** ✅ 已完成

`desktop.rs:1428-1475` — `open_notepad_window_now`：
```rust
let _ = window.show();          // L1474
let _ = window.set_focus();     // L1475 - 紧接 show() 后调用
```

Tauri v2 的 `show() → set_focus()` 在 Renderer 进程就绪前调用时，可能在 Windows 上触发焦点死锁。`show()` 是异步操作，在窗口内容真正渲染完成前调用 `set_focus()` 会阻塞消息循环。

**根本原因（推断）**：`show()` + `set_focus()` 之间无渲染完成等待。窗口 manager 在 Renderer 未就绪时尝试 focus，导致窗口进入"显示但无响应"状态。

**解决方案**：
- A（快速）：将 `set_focus()` 延迟，通过 `emit("notepad:activate")` 事件让前端在 `mounted` 后主动 `focus()`
- B（稳健）：`window.show()` 后 `tokio::time::sleep(100ms)`，实验性质等待渲染完成

**影响分析**：
- A 方案：改动 1 行 Rust + 1 行前端，无 UI 抖动
- B 方案：100ms 延迟用户可感知，不是最佳方案

**可行性** ⭐⭐⭐⭐：
**合理性** ⭐⭐⭐⭐：在渲染完成前 focus 是跨平台 GUI 开发的已知反模式
**必要性** 🔴 CRITICAL：9 条评论表明问题影响面广，应用完全无响应是严重 bug

**修复步骤**：
1. 移除 `open_notepad_window_now` 中的 `set_focus()` 调用
2. 在 `NotePad.tsx` 的 `useEffect` 中监听 `notepad:activate` 事件，调用 `window.focus()`
3. 在 Windows 上验证多次小窗打开/关闭无卡死

### 🟠 P1-1 — #297：txt 文件 UTF-8 编码检测

**源码查证** ✅ 已完成

`lib.rs:105-112` — `read_external_file`：
```rust
std::fs::read_to_string(&valid_path)
```
仅支持 UTF-8。中文 Windows 默认 GBK 编码的 `.txt` 文件直接失败。

**解决方案**：使用 `encoding_rs` 库（Rust 生态标准编码库）

```rust
fn read_file_with_fallback(path: &Path) -> Result<String, AppError> {
    let bytes = std::fs::read(path)?;
    if let Ok(s) = std::str::from_utf8(&bytes) {
        return Ok(s.to_owned());
    }
    // 尝试 GBK / GB18030
    let (cow, _, _) = encoding_rs::GBK.decode(&bytes);
    Ok(cow.into_owned())
}
```

**影响分析**：
- 功能：中文 Windows 用户可直接打开 txt 笔记
- 注释：需在依赖中添加 `encoding_rs = "0.8"`
- 副作用：`encoding_rs` 约 300KB，对桌面应用可忽略

**可行性** ⭐⭐⭐⭐⭐：3 行代码 + 1 个依赖
**合理性** ⭐⭐⭐⭐⭐：编码兼容是文件打开的基础功能
**必要性** 🟠 MAJOR：中文 Windows 用户完全无法打开 GBK 编码文件

**修复步骤**：
1. `Cargo.toml` 添加 `encoding_rs = "0.8"`
2. 在 `lib.rs` 添加 `read_file_with_encoding_fallback` 辅助函数

1. 录制时 Alt 键的捕获逻辑（Alt 在浏览器中有时触发菜单，`event.preventDefault()` 可能不生效）
2. 快捷键冲突检测（`global_shortcut_check`）返回 false 但 UI 无错误提示

**解决方案**：
- 增强错误提示：当 `checkGlobalShortcut` 返回 `available: false` 时显示明确错误
- 优化 Alt 键录制：在录制模式下更激进地 `preventDefault`

**可行性** ⭐⭐⭐⭐：
**必要性** 🟠 MAJOR：快捷键是核心功能入口

**修复步骤**：
1. 在 `handleStartRecording` 增加 Alt 键的 `preventDefault`
2. 在 `finishRecording` 中，检查 `checkGlobalShortcut` 结果，失败时显示具体错误
3. 添加"恢复默认"按钮的确认逻辑（而非静默无效）

### 🟡 P2-1 — #299：代码块复制按钮随滚动条移动（v1.1.0）

**源码查证** ⏳ CSS 级别问题

`MarkdownPreview.tsx:29-43` — `CodeBlock` 组件：
```tsx
<pre className="... overflow-x-auto relative group">
    <button className="absolute top-2 right-2 ...">
```

`position: relative` 在 `<pre>` 上，`position: absolute` 的按钮锚定在 `pre` 而非滚动容器。长代码横向滚动时按钮跟随移动。

**解决方案**：将复制按钮的锚定容器从 `pre` 改为外层 `div`（非滚动容器）

**可行性** ⭐⭐⭐⭐⭐：CSS 一行修改
**合理性** ⭐⭐⭐⭐⭐：明确 UI bug
**必要性** 🟡 MINOR：视觉问题，不影响功能

**修复步骤**：将 `relative` 移到包裹 `pre` 的外层容器，按钮 `absolute` 锚定在外层

### 🟡 P2-2 — #224：WSL 中笔记目录文件移除报错

**源码查证** ⚠️ 路径格式兼容问题

**根本原因**：`validate_path` 要求绝对路径，WSL 路径 `\\wsl.localhost\Ubuntu\...` 是 UNC 路径，`is_absolute()` 返回 `true`。但 `fs::metadata` 对 UNC 路径的行为不一致

**解决方案**：确认 WSL 路径在 Windows 上的 `std::fs` 操作是否支持

**可行性** ⭐⭐⭐：需 WSL 环境验证
**必要性** 🟡 MINOR：WSL 特殊场景

**修复步骤**：
1. 在 WSL 环境中测试 `fs::metadata` 对 `\\wsl.localhost\` 路径的行为
2. 如有必要，转换为 `\\wsl$\` 格式（Windows 原生路径转为 WSL 路径的映射）

**修复步骤**：CSS 级别修复，为 `.katex-display` 添加 `min-height` 或 KaTeX 配置中注入空 span
