## T1 — CSP 内容安全策略

### 修改内容

`src-tauri/tauri.conf.json` 的 `app.security.csp`：

```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com
```

### 审查发现

| # | 问题 | 严重度 | 说明 |
|---|------|:------:|------|
| 1 | **缺少 `img-src` 指令** | 🟠 MAJOR | 图片默认回退到 `default-src 'self'`。但花笺使用 Tauri `asset://` 协议加载本地图片（`images_save` / `images_get_base_dir`），`asset://localhost/` 是否被 `'self'` 匹配需要验证。如果 `asset://` 不被视为 `'self'`，所有本地图片将无法在预览中加载。 |
| 2 | **缺少 `font-src` 指令** | 🟡 MINOR | 花笺使用 `HarmonyOS_Sans_SC.ttf` 和 `SourceHanSerifSC-Regular.woff2` 本地字体。默认回退到 `default-src 'self'` 应可加载本地字体，但如果有外部字体 CDN 需求则需显式声明。 |
| 3 | `style-src https://cdn.jsdelivr.net` 范围过宽 | 🟡 MINOR | KaTeX CSS 从 jsdelivr 加载，但允许整个 jsdelivr CDN 的样式可能会被恶意 npm 包利用。建议限定到 `https://cdn.jsdelivr.net/npm/katex@`。 |
| 4 | `connect-src` 中 `https://mirrorchyan.com` 使用 HTTP 明文 | 🟡 MINOR | GitHub API 使用 HTTPS，但 MirrorChyan 应同样强制 HTTPS（目前配置是 https，确认无误）。✅ 实际上已使用 https。 |

### 建议修复

```
img-src 'self' asset: data:;
font-src 'self';
```

将 `style-src` 的 CDN 路径收窄为 `https://cdn.jsdelivr.net/npm/katex@0.16`。

## T2 — 路径遍历防护（validate_path）

### 修改内容

`src-tauri/src/lib.rs:16-46`，为 7 个文件 IO 命令增加 `validate_path` 校验。

### 审查发现

| # | 问题 | 严重度 | 说明 |
|---|------|:------:|------|
| 1 | **未调用 `canonicalize()`** | 🟠 MAJOR | 注释声称"`canonicalize()` 成功（路径真实存在且可解析）"（L13-14），但实际代码（L45）只是 `Ok(path_buf)` 直接返回，未做任何解析。这意味着 `C:\Users\winHP\..\..\Windows\System32\config\SAM` 这样的绝对路径中包含 `..` 会通过校验。 |
| 2 | **无 data_dir 边界限制** | 🟠 MAJOR | 用户可传入任意绝对路径（如 `C:\Windows\System32\drivers\etc\hosts`），`read_external_file` 会直接读取并返回内容。虽然 Tauri 命令只能由前端调用，但在本地应用中用户已拥有文件系统权限，此限制的实战意义有限。对于 Tauri 的 `assetProtocol.scope` 之外的路径，CSP 已提供额外防护。 |
| 3 | `read_external_file` 无存在性检查 | 🟡 MINOR | 文件不存在时，`std::fs::read_to_string` 返回 `io::Error`，错误消息对用户不友好（英文技术消息）。 |
| 4 | 7 个命令中 `notes_import_markdown` 和 `images_save_from_path` 需要源文件存在 | 🟡 MINOR | `validate_path` 不检查文件是否存在（注释说"导出场景除外"），但这 2 个命令的调用方预期源文件存在。不存在的路径会被下游 `fs` 操作捕获，但错误延迟到更深层，诊断不便。 |

### 建议修复

1. 对 `read_external_file`、`get_file_modified_time`、`notes_import_markdown`、`images_save_from_path` 这些需要源文件存在的命令，在 `validate_path` 后增加 `canonicalize` 和存在性检查
2. 对 `save_external_file` 和 `notes_export_markdown`（写入目标可能不存在），做父目录规范化但不要求文件存在

### T1 — CSP 内容安全策略

**合理性** ⭐⭐⭐⭐⭐
CSP 是 WebView 安全的第一道防线。项目原先 `csp: null` 使前端 XSS 可获得完整 Tauri 命令调用能力。Tauri v2 的 CSP 机制是标准且成熟的安全基础设施，使用它的决策完全正确。

**必要性** ⭐⭐⭐⭐⭐
绝对必要。原始 `csp: null` 是严重安全漏洞（安全审查标记为 CRITICAL）。一个无 CSP 的 Tauri WebView 等价于操作系统级的任意代码执行入口。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| 硬编码 CSP 字符串在 `tauri.conf.json` | 通过 `App::security_headers()` 动态设置 | ❌ 动态设置更灵活但更复杂，对接 Tauri 的事件生命周期。对当前规模过度设计。硬编码更简单可靠。 |
| CSP 字符串手工拼接 | 使用 `helmet`/CSP builder 库生成 | ❌ Tauri 使用原生 WebView CSP，不支持 npm CSP builder。 |
| `style-src` 放行整个 jsdelivr CDN | 限定到 `cdn.jsdelivr.net/npm/katex@0.16` | ✅ **建议采纳**。当前范围过宽，jsdelivr 上可托管任意内容。 |
| 无 `img-src` | 显式添加 `img-src 'self' asset:` | ✅ **建议采纳**。避免 `default-src` 回退的歧义。 |
| 无 `font-src` | 显式添加 `font-src 'self'` | ⚠️ 可选。本地字体通常被 `default-src 'self'` 覆盖。 |

**综合评价**：方向正确，细化不够。两个指令缺失 + 一个 CDN 范围过宽。

### T2 — 路径遍历防护

**合理性** ⭐⭐⭐⭐
思路正确——在前端到后端的 IO 边界处统一拦截。7 个命令共享同一个 `validate_path` 避免了重复代码。但实现有两个显著缺陷（见下）。

**必要性** ⭐⭐⭐⭐⭐
绝对必要。7 个命令全部接受前端传入的路径字符串且直接对接文件系统 API。安全审查标记为 CRITICAL，无路径校验意味着 XSS → 完整文件系统访问。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| 函数级手动调用 `validate_path` | **Tauri Command Guard（中间件模式）** | ⚠️ **值得讨论**。Tauri v2 不支持 command-level middleware，但可以包装为 `SafePath(String)` newtype + `FromCommand` 自动验证。好处：无法遗忘加校验。坏处：复杂度显著增加，且 7 个命令差异大（有的需要文件存在，有的不需要）。**当前的手动调用方案更务实**。 |
| 每个命令独立调用 `validate_path` | **宏（macro）自动注入** | ⚠️ 可行但过度。Rust 宏可自动为带 `path: String` 参数的命令注入校验。但如果将来需要按命令定制校验逻辑，宏会成为障碍。 |
| `is_absolute()` 作为唯一规范化手段 | **`canonicalize()` + 前缀限制（data_dir）** | ✅ **建议采纳**。至少追加 `canonicalize()` 消除 `..`。`data_dir` 边界限制可在安全审计意义上提供更强的"不离开沙箱"保证，但当前实战价值有限（本地应用用户已有文件系统权限）。 |
| 7 个命令逐一添加 `validate_path` | — | ✅ 实现正确且全面，无一遗漏。 |

**综合评价**：防御性拦截的思路和覆盖面正确，但实现有留白（未 canonicalize、无边界）。不是错误，是"差一步完美"。

### T3 — 禁用 sanitize style

**合理性** ⭐⭐⭐⭐⭐
`style` 属性暴露 CSS 注入面（`background-image:url(exfil)`, `position:fixed` phishing 等）。从 sanitize schema 移除 `style` 是标准做法——`rehype-sanitize` 的 `defaultSchema` 允许 `style` 但官方文档建议根据场景移除。

**必要性** ⭐⭐⭐⭐⭐
绝对必要。安全审查标记为 CRITICAL。`rehype-raw` + `style` 开放 = 可注入的 CSS 攻击向量。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| 覆盖 `attributes["*"]` 排除 `style` | **使用 CSS 属性白名单**（如 `style: ["color"]`） | ⚠️ 更精细但风险更高。白名单需要穷举所有安全 CSS 属性，容易遗漏。当前的全量禁用更安全，且花笺 Markdown 笔记不应依赖内联样式。✅ 当前方案更优。 |
| 仅影响 `rehypePluginsWithHtml` | **统一也为 `rehypePluginsDefault` 添加 sanitize** | ⚠️ **上游遗留问题**。当前 `renderHtml=false` 模式下完全不加载 `rehypeSanitize`，意味着该模式下 raw HTML 不被过滤。这不属于 T3 的施工范围，但在系统层面是安全缺口。建议在文档中标注。 |
| 手工修改 schema 属性表 | — | ✅ 实现简洁，一行删除完成，不影响其他功能。 |

**综合评价**：实现极简且正确。唯一的"不够"是 sanitize 只保护了 `renderHtml=true` 路径——但这是上游架构限制，不是本修复的问题。

### T4 — React Error Boundary

**合理性** ⭐⭐⭐⭐
React Error Boundary 是防止整棵树崩溃的标准模式。花笺有 79 个 React 组件且 `MarkdownPreview` 存在大量不安全的类型转换，防御性措施合理。

**必要性** ⭐⭐⭐
中等必要。Error Boundary 捕捉的是"意外错误"，如果代码质量高、测试覆盖好，它的触发概率极低。但花笺作为生产应用，且上游代码历史中确有不少类型不安全操作，**风险足够大，值得做**。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| 手写 Class Component（39 行） | **使用 `react-error-boundary` 库** | ⚠️ 第三方库提供了 `FallbackComponent`、`onReset`、`resetKeys` 等便利 API。但为 39 行的组件引入一个依赖是过度设计。✅ 当前方案更务实。 |
| 仅包裹 `<App />` 根节点 | **多层级 Error Boundary + 分段 fallback** | ⚠️ 理想做法是在 `MainWindow`、`NotePad`、`TileShowcase` 各自包裹独立的 Error Boundary，崩溃时只影响单个视图。但实现成本显著增加且 T4 标记为 MAJOR 而非 CRITICAL，根级覆盖已达成主要目标。 |
| fallback 仅显示静态文本 | **fallback 添加 "重新加载" 按钮** | ✅ **建议采纳**。纯文本"请重启"对普通用户不友好。`window.location.reload()` 一行即可。 |
| `componentDidCatch` 仅 log 到 console | **通过 Tauri invoke 写入错误日志文件** | ⚠️ 生产环境有价值，但属于锦上添花。 |

**综合评价**：实现克制且方向正确。fallback UX 和错误持久化可以更好，但不是严重问题。

### T5 — Cargo.lock 提交

**合理性** ⭐⭐⭐⭐⭐
Rust 官方明确建议：二进制应用（binary）提交 `Cargo.lock`，库项目不提交。花笺是二进制桌面应用，应当提交。这是 Rust 生态的共识。

**必要性** ⭐⭐⭐⭐
必要。不提交的风险：CI 构建使用与本地不同的传递依赖版本 → 构建不可复现 → 供应链审计盲区。已有案例中由依赖版本漂移导致的 CI 失败并不少见。

**更优实现**

无。这是 Rust 生态标准实践，没有替代方案可讨论。Cargo.lock 已在初始提交 `6eaac5b` 中自动追踪，无需额外操作。

### T7 — 统一版本号

**合理性** ⭐⭐⭐⭐
三处版本号不一致（`Cargo.toml` 落后为 `1.0.4`）是发布流程的可见缺陷。花笺的 `scripts/set-version.mjs` 已提供自动化同步工具，问题根源在于该脚本未被正确纳入发布流程。

**必要性** ⭐⭐
低。版本号不一致不影响编译产物（`Cargo.toml` 的 `version` 仅用于 `CARGO_PKG_VERSION` 宏和包元数据）。但对用户显示的"关于"页面版本和包管理器的版本标签会出现分歧。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| 手动修改 `Cargo.toml` | **发布流程绑定 `set-version.mjs`** | ✅ **建议采纳**。当前已有脚本但未强制。应在 `package.json` scripts 中确保 `version:sync` 在发布前执行（如加入 `preversion` hook）。 |
| 三处独立维护版本号 | **单一数据源（如 workspace Cargo.toml + node 读取）** | ⚠️ 过度。三处版本号服务于不同生态（npm、Cargo、Tauri），强行统一数据源引入非标准工具链。当前 `set-version.mjs` + 三个声明位置是合理的多生态实践。 |

**综合评价**：问题小而修复简单，但需强化流程层面避免复发。

---

### T8 — 替换 execCommand

**合理性** ⭐⭐⭐⭐
`document.execCommand` 已被 W3C 标记为废弃。花笺使用它的 4 处场景中，2 处（`insertText`）有明确现代替代方案，2 处（`undo`/`redo`，contentEditable）无标准替代 API。

**必要性** ⭐⭐
低。虽然 API 已废弃，但主流浏览器（Chrome/Firefox/Safari）不会在短期内移除 `execCommand`——太多旧网站依赖它。此修复的价值在于"未来兼容性"和"代码清洁度"，而非解决当前可观测的 bug。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| `undo`/`redo` 保留 execCommand | **自行实现 undo/redo 栈** | ❌ 过度。为 textarea 实现完整 undo 栈是数百行代码的工作，且需要处理所有编辑操作（键盘输入、粘贴、拖拽）。浏览器原生 undo 虽然有 API 兼容风险，但短时间内不可替代。✅ 当前方案正确。 |
| `useImagePaste.ts` 直接操作 `textarea.value` + `dispatchEvent("input")` | **通过 React state setter 操作** | ⚠️ `dispatchEvent("input")` 是一种"欺骗 React 受控组件"的模式——直接修改 DOM value 但通知 React 数据已变。更干净的做法是将图片链接插入逻辑提升到 React 状态层，由 `<NotePad>` 组件通过 prop 回调统一处理文本插入。但重构成本高于当前方案的风险。✅ 当前方案务实。 |
| 保留 `insertText` 但加注释 | 当前方案：直接删除 + `setContent(result)` | ✅ 更优。`MainWindow.tsx` 中 `execCommand("insertText")` 后紧跟 `setContent(result)`，删除前者无功能损失。 |

**综合评价**：务实的分层处理——能删则删，不能删则标记。`dispatchEvent("input")` 模式有改进空间但非紧急。

### T9 — Scheduler 优雅关闭

**合理性** ⭐⭐⭐
`AtomicBool` + 循环检查是线程优雅关闭的标准模式。实现正确且简洁。但问题本身的实战影响极小——进程退出时 OS 保证回收所有线程资源，`poll_auto_check` 中的 I/O 操作（HTTP 请求）在进程退出时自然中断。

**必要性** ⭐
极低。T9 属于"理论正确性"，非"实践必需性"。`thread::sleep` 阻塞中的线程在 OS 强制终止时不会导致数据损坏（因为文件锁和 HTTP 连接都由 OS 管理）。原始的无优雅关闭代码在实践中不会造成任何观测到的问题。但作为工程素养——优雅关闭是正确的做法。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| `AtomicBool` + `Relaxed` | `AtomicBool` + `Acquire/Release` | ⚪ 无差异。单标志位 + 单写入者 + 单读取者场景，`Relaxed` 足够。 |
| `thread::sleep` 阻塞 → 最长 60s 退出延迟 | **使用 `Condvar::wait_timeout`** | ⚠️ 可以让 `stop_auto_check_scheduler()` 即时唤醒线程。但 `Condvar` + `Mutex` 比 `AtomicBool` 复杂，且退出延迟在应用关闭场景下对用户不可见。✅ 当前方案简单够用。 |
| 裸 `thread::spawn` 不存储 `JoinHandle` | 存储 `JoinHandle` + 在 `stop` 中 `join` | ⚠️ 更完整但当前无实际需求。`reset_scheduler_shutdown()` 目前仅用于测试，如果未来需要确保旧线程退出后再启动新线程，才需要 `JoinHandle`。 |
| 在 `RunEvent::Exit` 中直接设置标志 | — | ✅ 正确位置。Tauri v2 的 `RunEvent::Exit` 在所有窗口关闭后触发。 |

**综合评价**：实现完美适合问题规模——没有过度设计也没有遗漏。问题是低优先级的，但解决方式是教科书级的。

### T10 — 安全扫描 CI

**合理性** ⭐⭐⭐⭐⭐
GitHub Actions + Dependabot + 三个扫描工具（`cargo-audit`、`npm audit`、OSV-Scanner）的组合覆盖了项目全部三个依赖生态系统。配置规范，触发条件合理。

**必要性** ⭐⭐⭐⭐
必要。已知漏洞（CVE）的自动化发现是供应链安全的基础设施。花笺作为桌面应用，依赖链长（Rust 传递依赖 + npm 传递依赖），手动追踪不现实。

**更优实现**

| 当前方案 | 替代方案 | 评价 |
|----------|----------|------|
| `npm audit --audit-level=high` | `--audit-level=critical` + 白名单文件 | ⚠️ `high` 级别可能在无可用修复时持续 CI 红色（false positive 阻塞）。`critical` 作为阻断门更务实。 |
| 三个独立 job（cargo-audit, npm-audit, osv-scanner） | 单 job 顺序执行 | ❌ 三个并行 job 是正确的：独立、快速失败、故障隔离。 |
| Dependabot 三个 ecosystem 分组 | — | ✅ 配置完整。npm / cargo / github-actions 三组，每周一更新，PR 上限 10。 |

**综合评价**：配置专业且克制。仅 `npm audit` 级别可微调。
