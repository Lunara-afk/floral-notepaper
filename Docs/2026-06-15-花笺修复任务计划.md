### T1: 设置 CSP 内容安全策略

- **优先级**: P0（安全阻断）
- **影响范围**: 1 文件 / 1 行
- **预计工时**: 0.5h
- **前置依赖**: 无

#### 现状

```json
// src-tauri/tauri.conf.json:27
"csp": null
```

#### 修复方案

```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com"
```

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全影响** | 🔒 大幅提升：WebView 无法加载外部脚本/资源，XSS 利用被限制 |
| **功能影响** | ⚠️ 如果应用中使用了外部 CDN（如 KaTeX 字体、JS 资源），需要在 CSP 中明确允许对应域名 |
| **性能影响** | 无（CSP 是 HTTP 头级别的声明，不影响运行时性能） |
| **兼容性** | 所有主流 WebView 均支持 CSP |
| **回滚难度** | 🟢 低 — 改回 `null` 即可 |

#### 验证方法

1. 构建后打开开发者工具 → 检查 Network 面板是否有被 CSP 阻止的资源
2. 导入包含 `<script>` 标签的恶意 .md 文件，确认脚本不执行
3. KaTeX 数学公式渲染正常（`style-src 'unsafe-inline'` 允许内联样式）

### T2: Tauri 命令路径遍历防护

- **优先级**: P0（安全阻断）
- **影响范围**: 1 文件 / 7 处命令
- **预计工时**: 3h
- **前置依赖**: 无

#### 受影响命令

| 命令 | 文件行号 | 操作 |
|------|----------|------|
| `read_external_file` | `lib.rs:67-73` | 读取任意文件 |
| `save_external_file` | `lib.rs:94-107` | 写入任意位置 |
| `get_file_modified_time` | `lib.rs:76-91` | 读取文件元数据 |
| `images_save_from_path` | `lib.rs:151-161` | 复制文件到应用目录 |
| `notes_import_markdown` | `lib.rs:50-59` | 导入 markdown |
| `notes_export_markdown` | `lib.rs:62-64` | 导出 markdown |
| `copy_background_image` | `lib.rs:188-223` | 复制背景图 |

#### 修复方案

**方案设计**: 根据命令用途分为两类处理——

**A. 导入/导出类**（`notes_import_markdown`, `notes_export_markdown`, `copy_background_image`）

> 这些命令的路径来自用户通过文件对话框选择（`dialog:allow-open`），理论上可信度较高。但仍需防线：
> - 用 `std::fs::canonicalize()` 解析真实路径
> - 检查解析后的路径是否为常规文件（排除符号链接和特殊文件）
> - 对导出路径，限制只能写到用户文档/桌面等已知目录

**B. 通用文件操作类**（`read_external_file`, `save_external_file`, `get_file_modified_time`, `images_save_from_path`）

> 这些命令的路径完全来自前端任意输入，风险最高：
> - 用 `std::fs::canonicalize()` 解析真实路径
> - 维护一个"允许访问的根目录"白名单（如 `data_dir`, 用户文档目录）
> - 校验 `canonicalize()` 结果必须以白名单目录为前缀
> - 拒绝符号链接（检查 `symlink_metadata`）

**示例（`read_external_file` 改造）**:
```rust
// 改造前
fn read_external_file(path: String) -> Result<String, AppError> {
    std::fs::read_to_string(&path).map_err(...)
}

// 改造后
fn read_external_file(path: String) -> Result<String, AppError> {
    let path = PathBuf::from(&path);
    // 1. 解析真实路径，拒绝符号链接
    let real = std::fs::canonicalize(&path).map_err(|e| AppError {
        code: "invalidPath".into(),
        message: format!("无法解析路径: {e}"),
        details: Default::default(),
    })?;
    // 2. 白名单校验（示例：只允许 data_dir 下的文件）
    let store = default_store()?;
    let allowed = store.data_dir().canonicalize()?;
    if !real.starts_with(&allowed) {
        return Err(AppError {
            code: "forbidden".into(),
            message: "不允许访问此路径".into(),
            details: Default::default(),
        });
    }
    std::fs::read_to_string(&real).map_err(...)
}
```

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全影响** | 🔒🔒 大幅提升：消除任意文件读写原语，XSS 无法逃逸到文件系统 |
| **功能影响** | ⚠️ **高**：这是改动最大的一项。文件对话框选择的路径必须被白名单覆盖。如果用户选择了白名单外的路径，该命令会返回错误 |
| **用户体验** | 导入/导出仍通过对话框进行，不受影响；`read_external_file` 需要确认所有调用场景 |
| **性能影响** | 微乎其微（`canonicalize()` 一次系统调用，毫秒级） |
| **兼容性** | Windows/macOS/Linux 均支持 `canonicalize()` |

#### ⚠️ 关键风险

- 如果前端有场景需要访问白名单外的系统路径（如读取系统字体、读取用户任意目录），需要在白名单中增加对应路径，或提供权限提升机制
- 建议先在 `read_external_file` 上试点，确认所有调用场景后再推广到其余 6 个命令

#### 验证方法

1. 单元测试：传入 `../../etc/passwd` 类路径，断言返回错误
2. 手动测试：通过文件对话框导入/导出文件，确认功能正常
3. 手动测试：调用 `read_external_file("/etc/passwd")`，确认被拒绝

### T3: 禁用 Markdown sanitize `style` 属性

- **优先级**: P0（安全阻断）
- **影响范围**: 1 文件 / 1 行
- **预计工时**: 0.5h
- **前置依赖**: 无

#### 现状

```typescript
// src/features/markdown/MarkdownPreview.tsx:78
"*": [
  ...(defaultSchema.attributes?.["*"] ?? []),
  "style",     // ← 删除此行
  "className",
],
```

#### 修复方案

**直接删除**:
```typescript
"*": [
  ...(defaultSchema.attributes?.["*"] ?? []),
  "className",
],
```

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全影响** | 🔒 消除 CSS 注入向量。`<span style="background-image:url(...)">` 将不再渲染 |
| **功能影响** | ⚠️ **中**：Markdown 渲染的 HTML 中内联 `style` 属性会被 `rehype-sanitize` 剥离。任何依赖内联样式的 markdown 内容将丢失颜色/字体样式 |
| **视觉效果** | 使用 `<font color="red">` 或 `<span style="color:red">` 的笔记内容样式会丢失 |
| **KaTeX** | 不受影响（KaTeX 使用 `class` 配合全局 CSS，不依赖内联 `style`） |

#### ⚠️ 缓解方案（如果需要保留部分样式能力）

如果用户确实需要在 Markdown 中染色文字，可以使用 CSS 过滤器方案代替直接删除：

```typescript
// 更细粒度的方案：仅允许安全的 CSS 属性
"*": [
  ...(defaultSchema.attributes?.["*"] ?? []),
  "className",
  // 不开放 "style"，改而允许 Markdown 扩展语法中的有限颜色
],
```

这需要评估：当前是否有用户使用内联样式。对于本地笔记应用，大多数用户使用纯 Markdown 语法，不依赖内联 `style`。

#### 验证方法

1. 创建一个笔记，在源码模式下输入 `<span style="background-image:url(https://evil.com)">test</span>`，切换到预览模式，确认背景图不加载、Network 无请求
2. 正常 Markdown 语法（`**粗体**`, `*斜体*`, `# 标题`）仍正常渲染

### T4: 添加 React Error Boundary

- **优先级**: P1（架构强化）
- **影响范围**: 新增 1 个组件 + `main.tsx` 1 处改动
- **预计工时**: 2h
- **前置依赖**: 无

#### 修复方案

1. **创建全局 Error Boundary** — 新增 `src/components/ErrorFallback.tsx`：
```tsx
import { Component, type ReactNode } from "react";

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
}

interface State {
  hasError: boolean;
  error: Error | null;
}

export class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false, error: null };

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error) {
    console.error("[ErrorBoundary] Uncaught error:", error);
  }

  render() {
    if (this.state.hasError) {
      return (
        this.props.fallback ?? (
          <div className="flex items-center justify-center h-screen text-ink-faint">
            应用遇到错误，请重启
          </div>
        )
      );
    }
    return this.props.children;
  }
}
```

2. **包裹 App 根组件** — 在 `main.tsx` 中：
```tsx
root.render(
  <ErrorBoundary>
    <App />
  </ErrorBoundary>
);
```

3. **可选：细粒度边界** — 为 `MarkdownPreview` 和 `Tile` 组件单独包裹，避免单个笔记渲染错误影响整个应用。

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全影响** | 🟡 间接：降低 DoS 风险（一个错误不再使整个应用崩溃） |
| **功能影响** | 🟢 正面：组件错误不再导致白屏，改为回退 UI |
| **性能影响** | 无（React 内置机制） |
| **用户体验** | 🟢 正面：遇到渲染错误时显示友好提示而非白屏 |
| **开发体验** | 🟢 正面：`componentDidCatch` 日志帮助定位错误 |

### T6: CDK token 改为 Header 传递

- **优先级**: P1（架构强化）
- **影响范围**: 1 文件 1 处 + MirrorChyan API 可能需要配合
- **预计工时**: 1.5h
- **前置依赖**: 确认 MirrorChyan API 是否支持 CDK 通过 Header 传递

#### 现状

```rust
// updater/check.rs:576-577
if let Some(cdk) = cdk.filter(|s| !s.trim().is_empty()) {
    url.query_pairs_mut().append_pair("cdk", cdk);
}
```

#### 修复方案

```rust
// 方式 1：HTTP Header
let mut request = client.get(url);
if let Some(cdk) = cdk.filter(|s| !s.trim().is_empty()) {
    request = request.header("X-MirrorChyan-CDK", cdk);
    // 或 "Authorization", "Bearer {cdk}" 取决于 API 约定
}
let response = request.send()...;
```

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全影响** | 🔒 token 不出现在 URL 中，不会被服务器访问日志或代理日志记录 |
| **功能影响** | ⚠️ **需要 MirrorChyan API 支持**。如果 MirrorChyan 目前只接受查询参数，需要先与上游沟通 |
| **兼容性** | 可能需要同时支持两种方式作为过渡 |
| **回退风险** | 🟡 中 — 需要 MirrorChyan 确认 Header 字段名和格式 |

#### ⚠️ 如果 MirrorChyan 暂时不支持 Header

替代方案：
1. 改用 POST 请求体传递 CDK（token 不出现在 URL）
2. 与 MirrorChyan 团队沟通增加 Header 支持
3. 短期缓解：确保错误日志中不包含 URL（当前已有测试保障）
