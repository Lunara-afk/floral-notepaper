## T1 — 设置 CSP 内容安全策略

### 基本信息

| 字段 | 内容 |
|------|------|
| **修复时间** | 2026-06-15 |
| **严重度** | 🔴 CRITICAL (P0) |
| **Phase** | Phase 1 — 安全阻断 |
| **状态** | ✅ 已完成 |

### 问题描述

`tauri.conf.json` 中 `"csp": null` 导致 WebView 完全不发送 Content Security Policy 头。结合 `rehype-raw` 允许原始 HTML 渲染，攻击者可通过恶意 `.md` 文件在 WebView 中执行任意 JavaScript，进而调用 Tauri 命令读写用户文件系统。

### 根本原因

Tauri v2 中 `csp: null` 表示显式禁用 CSP。项目在开发过程中未设置 CSP，一直沿用 null 值发布到生产环境。

### 修复方案

将 `null` 替换为严格 CSP 策略：

```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com
```

策略说明：
- `default-src 'self'` — 所有资源默认仅允许同源
- `script-src 'self'` — 仅允许同源脚本（禁止内联脚本和外部脚本）
- `style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net` — 允许同源样式 + 内联样式（KaTeX 需要）+ KaTeX CDN
- `connect-src 'self' https://mirrorchyan.com https://api.github.com` — 仅允许连接到更新服务器

### 影响分析

| 影响维度 | 分析 |
|----------|------|
| **安全性** | 🔒🔒🔒 大幅提升：WebView 无法加载外部脚本，XSS 攻击面从"完整系统访问"缩小为"同源限制" |
| **功能兼容性** | KaTeX CDN 样式已在 CSP 中放行，数学公式渲染不受影响 |
| **副作用** | 如果应用中有其他外部资源加载需求，会被 CSP 阻止。当前检查确认仅 KaTeX CDN 需要 |

### 改动详情

| 文件 | 行号 | 改动前 | 改动后 |
|------|------|--------|--------|
| `src-tauri/tauri.conf.json` | L29 | `"csp": null` | `"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com"` |

### 验证方式

1. 构建后打开开发者工具 → Network 面板 → 确认无 CSP 违规报告
2. 导入包含 `<script>alert(1)</script>` 的恶意 .md 文件 → 脚本不执行
3. KaTeX 数学公式（`$$E=mc^2$$`）正常渲染

## T2 — Tauri 命令路径遍历防护

### 基本信息

| 字段 | 内容 |
|------|------|
| **修复时间** | 2026-06-15 |
| **严重度** | 🔴 CRITICAL (P0) |
| **Phase** | Phase 1 — 安全阻断 |
| **状态** | ✅ 已完成 |

### 问题描述

7 个 Tauri 命令接受前端传入的任意路径字符串，直接调用文件系统 API（`read_to_string`、`write`、`copy`、`metadata` 等），无任何路径合法性校验。结合 CSP 被禁用的历史状态，前端 XSS 可获得完整的文件系统读写能力。

### 根本原因

开发时假设文件路径来自文件对话框（可信来源），未考虑前端可能构造恶意路径（如 `../../etc/passwd`）的场景。

### 修复方案

1. **新增 `validate_path()` 辅助函数**，对所有 7 个命令的路径参数进行统一校验：
   - 路径不可为空
   - 路径不可包含空字节（防止 C 风格截断攻击）
   - 路径必须为绝对路径（防止相对路径遍历，如 `../../etc/passwd`）

2. **逐项修改 7 个命令**，将 `PathBuf::from(path)` 替换为 `validate_path(&path)?`

### 受影响的 7 个命令

| 命令 | 行号 | 操作类型 | 风险描述 |
|------|------|----------|----------|
| `notes_import_markdown` | L50-59 | 导入 markdown | 任意文件读取 |
| `notes_export_markdown` | L62-64 | 导出 markdown | 任意位置写入 |
| `read_external_file` | L67-73 | 读取外部文件 | 任意文件读取 |
| `get_file_modified_time` | L76-91 | 获取文件时间 | 元数据泄露 |
| `save_external_file` | L94-107 | 保存外部文件 | 任意位置写入 |
| `images_save_from_path` | L151-161 | 导入图片 | 任意文件复制 |
| `copy_background_image` | L188-223 | 复制背景图 | 任意文件复制 |

### 影响分析

| 影响维度 | 分析 |
|----------|------|
| **安全性** | 🔒🔒 大幅提升：相对路径遍历被阻断，空字节注入被防护 |
| **功能兼容性** | 文件对话框返回的路径均为绝对路径，不受影响 |
| **副作用** | 如果前端曾向这些命令传入相对路径，现在会收到 `invalidPath` 错误。当前代码中未发现此场景 |

### 改动详情

| 文件 | 行号范围 | 改动说明 |
|------|----------|----------|
| `src-tauri/src/lib.rs` | L9-44 | 新增 `validate_path()` 函数（37 行） |
| `src-tauri/src/lib.rs` | L88-89 | `notes_import_markdown` → `validate_path` |
| `src-tauri/src/lib.rs` | L97-98 | `notes_export_markdown` → `validate_path` |
| `src-tauri/src/lib.rs` | L103-104 | `read_external_file` → `validate_path` |

1. 正常通过文件对话框导入/导出 → 功能正常

3. 前端尝试调用 `read_external_file("C:\Users\...")`（合法绝对路径）→ 正常读取

## T3 — 禁用 Markdown sanitize style 属性

### 基本信息

| 字段 | 内容 |
|------|------|
| **修复时间** | 2026-06-15 |
| **严重度** | 🔴 CRITICAL (P0) |
| **Phase** | Phase 1 — 安全阻断 |
| **状态** | ✅ 已完成 |

### 问题描述

`MarkdownPreview.tsx` 中的 `rehype-sanitize` schema 在所有元素 (`"*"`) 上显式允许了 `"style"` 属性。当 `renderHtmlMarkdown` 选项开启时，攻击者可在 markdown 中通过内联 style 实现 CSS 注入，如 `background-image:url(...)` 数据外泄。

### 根本原因

最初可能为了支持 `<span style="color:red">` 等内联样式，在 sanitize schema 中开放了 `style` 属性，但未评估其安全风险。

### 修复方案

从 `"*"` 的属性列表中删除 `"style"`，`rehype-sanitize` 将自动剥离所有元素的 style 属性。

### 影响分析

| 影响维度 | 分析 |
|----------|------|
| **安全性** | 🔒 消除 CSS 注入向量。`background-image:url()`, `position:fixed` 等 CSS 攻击无效 |
| **功能兼容性** | Markdown 笔记中通过 HTML `<span style="...">` 设置的内联样式将不再渲染 |

### 改动详情

| 文件 | 行号 | 改动前 | 改动后 |
|------|------|--------|--------|
| `src/features/markdown/MarkdownPreview.tsx` | L78 | `"style",` | （已删除） |

### 验证方式

1. 创建笔记，源码模式输入 `<span style="background-image:url(https://evil.com/steal)">test</span>`，预览模式确认背景图不加载

## T8 — 替换 document.execCommand

### 基本信息

| 字段 | 内容 |
|------|------|
| **修复时间** | 2026-06-15 |
| **严重度** | 🟡 MINOR (P2) |

### 问题描述

项目中 4 处使用了已弃用的 `document.execCommand` API，未来浏览器可能彻底移除该 API：

1. `MainWindow.tsx:274` — `execCommand("insertText", ...)` 插入文本
2. `MainWindow.tsx:285` — `execCommand(command)` undo/redo
3. `useImagePaste.ts:53` — `execCommand("insertText", ...)` 插入图片链接
4. `ContextMenu.tsx:164` — `execCommand(command)` contentEditable 回退

### 根本原因

`execCommand` 是历史遗留 API，在现代应用中应迁移到 `navigator.clipboard` 或直接值操作。

### 修复方案

- **第 1 处（MainWindow.tsx:274）**：已有 `setContent(result)` 紧随其后，直接删除 `execCommand` 调用
- **第 2 处（MainWindow.tsx:285）**：`undo`/`redo` 无程序化替代 API，保留并加注释说明
- **第 3 处（useImagePaste.ts:53）**：改为直接操作 `textarea.value` + `dispatchEvent("input")`
- **第 4 处（ContextMenu.tsx:164）**：contentEditable 元素的唯一可用方案，保留并加注释

### 影响分析

## T9 — Scheduler 线程优雅关闭

### 基本信息

| 字段 | 内容 |
|------|------|
| **修复时间** | 2026-06-15 |
| **严重度** | 🟡 MINOR (P2) |
| **Phase** | Phase 3 — 体验优化 |
| **状态** | ✅ 已完成 |

### 问题描述

`updater/scheduler.rs` 中 `start_auto_check_scheduler` 函数启动一个 `thread::spawn`，内部使用 `loop {}` 无限循环轮询更新检查。应用退出时线程被操作系统强制终止，可能处于 I/O 中间状态，存在极小概率的数据损坏风险。

### 根本原因

原始实现未考虑优雅关闭机制。因为 Tauri 退出时操作系统会强制终止所有子线程，功能上无问题，但不满足正确性要求。

### 修复方案

1. **添加静态关闭标志**：`static SCHEDULER_SHUTDOWN: AtomicBool`，在循环首尾各检查一次
2. **添加 `stop_auto_check_scheduler()` 公开函数**
3. **在 `lib.rs` 的 `RunEvent::Exit` 中调用**：应用退出前设置标志，让线程在下一次循环迭代时自然退出

### 影响分析

| 影响维度 | 分析 |
|----------|------|
| **正确性** | ✅ 线程在退出前有机会完成当前 I/O 操作 |
| **性能** | 每个循环额外两次 `Ordering::Relaxed` 原子读，微乎其微 |
| **副作用** | 无 |

### 改动详情

| 文件 | 行号 | 改动说明 |
|------|------|----------|
| `src-tauri/src/updater/scheduler.rs` | L3 | 新增 `use std::sync::atomic::{AtomicBool, Ordering}` |
| `src-tauri/src/updater/scheduler.rs` | L11-24 | 新增 `SCHEDULER_SHUTDOWN` 静态变量、`stop_auto_check_scheduler()`、`reset_scheduler_shutdown()` |
| `src-tauri/src/updater/scheduler.rs` | L27,46-49 | 循环中添加两次 `SCHEDULER_SHUTDOWN` 检查 |
| `src-tauri/src/lib.rs` | L486-488 | 在 `RunEvent::Exit` 分支中调用 `stop_auto_check_scheduler()` |

### 验证方式

1. 修改 `INITIAL_DELAY` 和 `POLL_INTERVAL` 为短时间（测试用），确认循环正常执行
2. 关闭应用，确认线程在下一个 sleep 后退出（可通过日志验证）

### P5-T1 — CSP 策略补充 img-src / font-src + 收窄 CDN 范围

#### 基本信息

| 字段 | 内容 |
|------|------|
| **补丁时间** | 2026-06-15 |
| **所属修复** | Phase 1 — T1 CSP 内容安全策略 |
| **触发原因** | 自审查报告：缺少 `img-src`（可能导致本地图片 `asset://` 协议图片不显示）、缺少 `font-src`、CDN 范围过宽 |

#### 问题描述

原始 CSP 策略存在三个不足：
1. **无 `img-src`**：图片资源回退到 `default-src 'self'`。花笺使用 Tauri `asset://` 协议加载本地图片（通过 `images_get_base_dir` + `convertFileSrc`），`asset://` 是否被 `'self'` 覆盖存在歧义
2. **无 `font-src`**：本地字体（HarmonyOS Sans SC、SourceHan Serif SC）虽可能被 `default-src 'self'` 覆盖，但显式声明更可靠
3. **`style-src https://cdn.jsdelivr.net` 范围过宽**：仅需 KaTeX CSS（`katex@0.16`），放行整个 jsdelivr CDN 意味着任何托管在该 CDN 上的样式文件都可被加载

#### 合理性与必要性

| 维度 | 评估 |
|------|------|
| **合理性** | ⭐⭐⭐⭐⭐ CSP 策略细化是标准的安全加固做法。显式指令比隐式回退更安全、更可审计 |
| **必要性** | 🟠 MAJOR。`img-src` 缺失可能导致本地图片加载失败，直接影响用户体验；CDN 范围过宽增加攻击面 |

#### 修复方案

将 CSP 字符串从：
```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com
```
改为：
```
default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net/npm/katex@0.16; img-src 'self' asset: data:; font-src 'self'; connect-src 'self' https://mirrorchyan.com https://api.github.com
```

变更点：
- `img-src 'self' asset: data:` — 允许同源图片 + `asset://` 协议（Tauri 本地图片）+ `data:` URI
- `font-src 'self'` — 显式允许同源字体加载
- `style-src` 从 `cdn.jsdelivr.net` 收窄为 `cdn.jsdelivr.net/npm/katex@0.16`

#### 影响分析

| 维度 | 分析 |
|------|------|
| **功能兼容性** | KaTeX 数学公式从 CDN `katex@0.16` 加载 CSS，路径收窄后不影响当前版本。版本升级需同步更新 CSP |
| **安全性** | CDN 攻击面从整个 jsdelivr 收窄到单一包路径。`img-src` 显式放行 `asset:` 协议 |
| **副作用** | 无。`data:` URI 允许内联图片（base64），关闭前此项被 `default-src` 隐式覆盖 |

#### 改动详情

| 文件 | 行号 | 改动 |
|------|:----:|------|
| `src-tauri/tauri.conf.json` | L29 | CSP 策略字符串增加 `img-src`、`font-src`、收窄 `style-src` |

#### 验证方式

2. 构建后打开开发者工具 → Network 面板 → 确认无 CSP 违规报告
3. KaTeX 数学公式（`$$E=mc^2$$`）正常渲染

### P5-T2 — validate_path 增加父目录遍历拦截 + canonicalize 分层

#### 基本信息

| 字段 | 内容 |
|------|------|
| **补丁时间** | 2026-06-15 |
| **所属修复** | Phase 1 — T2 Tauri 命令路径遍历防护 |
| **触发原因** | 自审查报告：注释声称调用 `canonicalize()` 但实际未实现；绝对路径中的 `..` 未被拦截 |
| **涉及文件** | `src-tauri/src/lib.rs`, `src-tauri/src/updater/mod.rs` |

#### 问题描述

原 `validate_path` 函数存在两个设计缺陷：

1. **注释与代码不一致**：L13-15 注释声称"canonicalize() 成功（路径真实存在且可解析）"，但 L45 只是 `Ok(path_buf)` 直接返回，从未调用 canonicalize
2. **父目录遍历未拦截**：绝对路径如 `C:\Users\..\..\Windows\System32\config\SAM` 通过 `is_absolute()` 检查后直接放行。`..` 组件在绝对路径中同样可以绕过预期文件范围

#### 合理性与必要性

| 维度 | 评估 |
|------|------|
| **合理性** | ⭐⭐⭐⭐⭐ 路径遍历防护的核心就是拦截 `..` 组件和解析 realpath。当前修复补齐了原始设计承诺的功能 |
| **必要性** | 🟠 MAJOR。虽然前端 XSS 已被 T1 CSP 拦截，但路径校验自身的一致性同样重要——防御不应依赖单一层次 |

#### 修复方案

采用**双层防护**架构：

**第 1 层 — `validate_path`（格式检查）**：新增 `..` 组件遍历拦截
```rust
for component in path_buf.components() {
    if component == std::path::Component::ParentDir {
        return Err(AppError { ... });  // "路径包含父目录遍历（..）"
    }
}
```
该层不要求文件存在，适用于所有 7 个命令（包括写入场景）。

**第 2 层 — `resolve_existing_path`（路径解析）**：新增辅助函数
```rust
fn resolve_existing_path(path: &PathBuf) -> Result<PathBuf, AppError> {
    path.canonicalize().map_err(|e| AppError {
        code: "invalidPath".into(),
        message: format!("路径无法解析: {e}"),
    })
}
```
该层 **需要文件存在**，适用于 5 个读命令：
- `notes_import_markdown` — 导入 markdown
- `read_external_file` — 读取外部文件
- `get_file_modified_time` — 获取文件时间
- `images_save_from_path` — 导入图片
- `copy_background_image` — 复制背景图

写入命令（`notes_export_markdown`、`save_external_file`）只使用第 1 层，因为目标文件尚不存在。

**修复路径调用**：`lib.rs` 中 `updater::scheduler::stop_auto_check_scheduler()` 使用的是私有模块路径，改为通过 `updater/mod.rs` 的 public re-export 调用。

#### 影响分析

| 维度 | 分析 |
|------|------|
| **安全性** | 🔒🔒 C:\Users\..\..\Windows 型路径遍历被 `..` 组件拦截阻断。Symlink/junction 类别名由 `canonicalize()` 解析 |
| **功能兼容性** | 5 个读命令需要文件实际存在，传导用户的预期行为：不存在的文件会收到清晰的 "路径无法解析" 错误，而非模糊的 "io" 错误 |
| **副作用** | 写入命令行为不变（不追加 canonicalize） |

### P6-1 — #38：开启小窗时卡死（移除 Rust 端 set_focus）

#### 基本信息

| 字段 | 内容 |
|------|------|
| **补丁时间** | 2026-06-15 |
| **上游 Issue** | [#38](https://github.com/Achilng/floral-notepaper/issues/38)（9 条评论，最早 2026-05-18） |
| **平台** | Windows |
| **涉及文件** | `src-tauri/src/desktop.rs`, `src/components/NotePad.tsx`（无需改动） |

#### 问题描述

用户开启小窗时出现类似覆盖全屏置顶模式，Alt+Tab 无响应，窗口外部的桌面可操作但应用本身不响应点击和键盘输入。9 条评论，多个用户复现。

#### 根本原因

`desktop.rs` 中两处 `show()` 后立即 `set_focus()`：
- `activate_pooled_notepad`（L1474-1475）：池化窗口复用路径
- `open_or_focus_window` 已有窗口分支（L1788-1789）

Tauri v2 的 `window.show()` 是异步操作——在 Windows 的 Renderer 进程完成首帧渲染前返回。WebView2 的 `set_focus()` 如果在 `WM_SHOWWINDOW` 消息处理完毕前被调用，会进入"窗口可见但消息循环挂起"的不可恢复状态。

PR #127（Lunara-afk）尝试通过 800ms 防抖缓解，2026-05-23 被 Achilng 以"不够优雅 + 牺牲手感"为由拒绝。

#### 修复方案

移除 Rust 层的两处 `set_focus()` 调用：

| 位置 | 改动 |
|:----|:----|
| `activate_pooled_notepad` L1475 | 删除 `let _ = window.set_focus();` |
| `open_or_focus_window` L1789 | 删除 `window.set_focus()?;` |

前端 `NotePad.tsx:279-294` 的 `notepad:activate` 事件处理器已包含 `showCurrentWindow()` 和 `contentRef.current?.focus()`，窗口显示和编辑区聚焦均由前端自主管理，无需 Rust 端干预。

#### 合理性与必要性

| 维度 | 评估 |
|------|------|
| **合理性** | ⭐⭐⭐⭐⭐ 在 Renderer 就绪前 focus 是跨平台 GUI 反模式。前端事件驱动比 Rust 端"猜测时序"更可靠 |
| **必要性** | 🔴 CRITICAL 9 条评论，应用完全无响应是严重 bug |

### P6-2 — #284：磁贴找不到导入的 txt 笔记（补充 #297 修复范围）

#### 基本信息

| 字段 | 内容 |
|------|------|
| **补丁时间** | 2026-06-15 |
| **上游 Issue** | [#284](https://github.com/Achilng/floral-notepaper/issues/284) |
| **所属修复** | Phase 6 + #297 补充 |
| **涉及文件** | `src-tauri/src/lib.rs`, `src-tauri/src/services/notes.rs` |

#### 问题描述

用户导入 txt 文件后启用磁贴（钉屏），显示"找不到此笔记"。

#### 根本原因

#297 的 GBK 编码修复（`read_file_with_encoding_fallback`）仅覆盖了 `read_external_file` 命令（从外部打开文件），**遗漏了 `import_markdown_file` 函数**（文件导入到应用内）。该函数仍使用 `std::fs::read_to_string`，导入 GBK 编码的 txt 时失败 → `Note` 对象未被创建 → 磁贴通过 `note_id` 查找窗口时找不到。

#### 修复方案

1. `lib.rs:129` — 将 `read_file_with_encoding_fallback` 从私有函数改为 `pub(crate)`，允许同 crate 内的 `notes.rs` 调用
2. `notes.rs:877` — 将 `import_markdown_file` 中的 `fs::read_to_string` 替换为 `crate::read_file_with_encoding_fallback`

#### 合理性与必要性

| 维度 | 评估 |
|------|------|
| **合理性** | ⭐⭐⭐⭐⭐ 编码回退逻辑应在所有读取文件内容的路径中一致使用 |
| **必要性** | 🟠 MAJOR 导入是笔记应用的核心入口，GBK 文件无法导入直接影响中文 Windows 用户 |
| **安全性** | ⚪ 无影响 |
| **可行性** | ⭐⭐⭐⭐⭐ 1 行可见性修改 + 1 行函数替换 |

#### 影响分析

| 维度 | 分析 |
|------|------|
| **功能** | GBK 编码的 .txt 文件可以被正常导入到花笺笔记中 |
| **磁贴** | 导入成功后 `Note` 对象被正确创建，磁贴可以找到对应笔记 |
| **副作用** | `pub(crate)` 暴露了内部函数但仅限同 crate，不影响 Tauri 命令边界 |

#### 改动详情

| 文件 | 行号 | 改动 |
|------|:----:|------|
| `src-tauri/src/lib.rs` | L129 | `fn read_file_with_encoding_fallback` → `pub(crate) fn read_file_with_encoding_fallback` |
| `src-tauri/src/services/notes.rs` | L877 | `fs::read_to_string(path)` → `crate::read_file_with_encoding_fallback(path)` |

### P6-3 — #129：块级公式无公式时序号渲染不完全

#### 基本信息

| 字段 | 内容 |
|------|------|
| **补丁时间** | 2026-06-15 |
| **上游 Issue** | [#129](https://github.com/Achilng/floral-notepaper/issues/129) |
| **涉及文件** | `src/App.css` |

#### 问题描述

块级公式仅输入 `\tag{1}` 不输入公式正文时，序号渲染不完整（只显示上半部分）。

#### 根本原因

KaTeX 渲染空公式时，生成的 DOM 节点中公式盒子高度为零。序号 `<span>` 以 `position: absolute` 定位在公式盒子内，零高盒子 → 序号被裁剪。这是 KaTeX 自身的渲染行为，非花笺代码缺陷。

#### 修复方案

在 `App.css:621` 的 `.katex-display > .katex` 规则中添加 `min-height: 1em`，确保即使公式为空，KaTeX 公式盒子也有最小高度。

#### 合理性与必要性

| 维度 | 评估 |
|------|------|
| **合理性** | ⭐⭐⭐⭐⭐ 一行 CSS 解决 KaTeX 渲染的边缘情况 |
| **必要性** | 🟡 MINOR 极端边缘情况（仅 `\tag{1}` 不带公式正文） |
| **安全性** | ⚪ 无影响 |

#### 影响分析

| 维度 | 分析 |
|------|------|
| **空公式** | `\tag{1}` 渲染的序号完整可见 |
| **正常公式** | `E=mc^2 \tag{1}` 不受影响——正常公式高度 > 1em |
| **副作用** | 无。`min-height: 1em` 对已有正常渲染的公式无影响 |

#### 改动详情

| 文件 | 行号 | 改动 |
|------|:----:|------|
| `src/App.css` | L621 | `.katex-display > .katex` 规则添加 `min-height: 1em;` |

#### 验证方式

1. 在花笺中新建笔记，输入 `$$\tag{1}$$`
2. 确认序号完整显示
